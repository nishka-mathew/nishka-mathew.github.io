01 — What It Does

Real-time trending movies, personal watchlists, and a design that gets out of the way.

CinemaQuest is an iOS app for people who love movies and hate losing track of them. It pulls live trending data from The Movie Database API, displays detailed information for each film, and lets users manage two personal lists: Up Next for films they plan to watch, and Watched for the ones they have. Users can also add movies that are not in the official database, leave notes, set watch dates, and sort the trending list by release date, popularity, or rating.

I built this as a solo project for my iOS Development course at UNC Charlotte, in Swift using Xcode and UIKit.

03 — Working with the TMDB API

Postman before Swift. Always figure out what the data looks like before you try to use it.

The whole app depends on live data from The Movie Database API, so understanding that API thoroughly was the foundation everything else was built on. Before writing a single line of Swift, I used Postman to explore and test the TMDB endpoints, inspecting the raw JSON responses, verifying the structure of the Movie object, and confirming exactly which fields were being returned and in what format.

That upfront work made the actual integration significantly cleaner. The Trending Movies endpoint returns a GET request with an array of Movie objects, each carrying an id, title, overview, poster path, release date, vote average, and popularity score. The Movie Detail endpoint returns a single Movie with the same shape, which kept the data model consistent across screens.

Knowing the shape of your data before you write the code that handles it is the difference between a smooth integration and a debugging session that takes the whole afternoon.

Async image loading for the movie posters is handled via Nuke, which fetches and caches poster images from the TMDB image base URL in the background so the trending feed scrolls without stuttering even while images are still loading.

04 — How It's Built

Five screens, one API, and a data model that holds everything together.

The app is organized around five screens: the Trending Movies feed, the Movie Detail view, the Up Next list, the Watched list, and the Add Movie flow that appears from both list screens.

The list screens use two custom model types: UpNextEntry, which pairs a Movie with an optional due date and note, and WatchedEntry, which pairs a Movie with a watched date and note. Both lists persist locally across sessions so users do not lose their data when they close the app.

05 — Design Decisions

Mobile first meant actually thinking about how people use their phones.

The bottom tab navigation was a deliberate choice. Users switch between Trending, Up Next, and Watched constantly, and making that a single thumb-tap rather than a back-button flow made the app feel significantly faster to use. The sort controls on the Trending screen sit above the list rather than inside a menu, so changing the sort order does not interrupt browsing.

The Add Movie flow supports both API-sourced movies and user-created ones, which came out of thinking about the kinds of films that would not be in a trending feed but would absolutely end up on someone's watchlist: older films, obscure releases, or things a friend recommended by word of mouth.

06 — What's Next

There is a lot of room to grow from here.

The foundation is solid and the scope was intentionally well-defined for a class project. The features I would add next are a recommendation engine based on viewing history, a social layer so users can see what friends are watching, and an expanded search that pulls from the full TMDB catalog rather than just the trending endpoint. A widget for the home screen showing your next unwatched film would also be genuinely useful.

Impact

A fully functional iOS app built solo in one semester.

Live API data tested and integrated via Postman, persistent watchlists, custom sorting, and a design that respects how people actually use their phones.
